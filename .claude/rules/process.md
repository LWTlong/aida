# Process Rules

- [RULE-002] 需求复杂时禁止直接开始编码。必须按顺序执行：1) 复述理解——逐条说明对需求的理解；2) 给出方案——列出实现思路、涉及文件、关键设计决策；3) 等待用户确认；4) 拆任务执行。判断"复杂"的标准：涉及多个文件改动、跨模块联动、API 字段变更、新增功能模式等。简单的单行修复、i18n 补充、样式调整可直接执行。
- [RULE-006] bug-fixer 执行时第 4 步（记录反馈）强制不可跳过，必须更新 bugTask.md
- [RULE-012] 禁止臆想。当需求描述不明确、或修改会影响现有行为时，必须先向用户确认，不得自行假设用户意图并动手修改。宁可多问一句，不可错改一处。
- [RULE-014] 禁止在未明确需求的情况下自行添加功能。AI 可以在实现完需求后主动提出"是否需要 X 功能"的建议，但不得直接将其实现进代码。擅自添加的功能会导致验收时行为与需求不符、出现用户未预期的接口调用或弹窗交互。
